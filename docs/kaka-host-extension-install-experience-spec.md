# Kaka Host Extension Install Experience Spec

Updated: 2026-06-11

This document records the product decision for the next Host Extension work:
Kaka should feel installable through the host runtime, not assembled by a user
from adapter commands.

Implementation status: P3.19 is implemented in Runtime Kit by strengthening the
existing `host-extension-install-package` output with host UI acceptance
metadata, generated `host-ui/acceptance.json`, ordered install-drill steps,
evidence receipt refs, TLS/readiness/evidence/Codex developer source release
gates, and static manifest/schema drift protection. P3.31 now extends that same
handoff with `ordinary_user_quickstart`, `host-ui/user-quickstart.md`, and
`install-drill/user-journey.json`. P3.35 now extends the same handoff with
`installation_blueprint` and `host-ui/installation-blueprint.json`:
`docs/superpowers/plans/2026-06-11-kaka-pocket-agents-host-native-installation-blueprint.md`.

Follow-up productization roadmap:

## Decision

The ordinary-user integration must be host-native:

1. Hermes ships Kaka Mobile Bridge as a Hermes Plugin.
2. OpenClaw ships Kaka Mobile Bridge as an OpenClaw Skill or sidecar package.
3. The user installs that package from a host-owned channel.
4. The user opens the host Kaka Mobile Bridge panel, explicitly enables the
   bridge, shows a short-lived QR or opts into Bonjour, then pairs Kaka iPhone.

Runtime Kit can generate contracts, package handoff materials, readiness
reports, conformance evidence, and developer automation source for host teams.
It must not become the public installer.

Codex plugin/skill automation is only for Hermes/OpenClaw engineers. It can
help scaffold, validate, and review the Host Extension package, but ordinary
users should not install Kaka by installing a Codex plugin or editing a Codex
skill.

## Follow-Up Development Decision

The next install-experience work should make the host-native path more concrete,
not introduce a second public installer. The desired product shape is:

1. Hermes/OpenClaw owns the Plugin/Skill package and native update channel.
2. The package installs disabled-by-default.
3. The host UI shows a Kaka Mobile Bridge panel with explicit Enable, Start,
   Show QR, Bonjour, Health, Revoke iPhone, Update, Logs, Repair, and Uninstall
   controls.
4. Runtime Kit generators provide host-team contracts, acceptance artifacts, and
   readiness receipts.
5. Codex plugin/skill automation remains source-only support for host engineers,
   not the ordinary-user onboarding flow.

P3.35 describes and validates package manifest expectations, host UI entry
points, disabled defaults, local TLS display, pairing controls, lifecycle
receipts, and release gates. It does not install, sign, publish, start the
bridge, bind LAN, advertise Bonjour, mint tokens, invoke private adapters, write
Codex plugin or skill roots, or change `/mobile/v1`.

Post-P3.35 rule: do not add another repository-only installation wrapper unless
new host-owned evidence requires a read-only extension to an existing
readiness/intake/blueprint contract. Without real Hermes/OpenClaw package facts,
the next useful repository work should be product-facing and separately
permissioned. P3.36a Inbox Voice Capture Context Copy and P3.36b Explicit
Paste-to-Inbox Courier are now complete, and P3.37 Inbox Result Review
Provenance is complete. P3.38 Explicit Files-to-Inbox Import is also complete;
the next blocked-period product work should be another separately scoped slice.

Follow-up install UX rule: the next installation deliverable is not a public
Codex plugin, a public Codex skill, or another command wrapper. It is a
host-native package candidate from Hermes/OpenClaw, reviewed locally through
P3.28 material intake and then exercised through P3.7 external install drill.
The candidate must include the host-native package ref, host UI entry point,
disabled-by-default install evidence, extension-internal adapter command
location, drill receipts, P3.2 conformance ref, P3.4 evidence manifest ref, and
ordinary-user release notes. If those materials are not available, keep
installation work paused and continue with a separately permissioned product
slice.

## P3.35 Implemented Scope

P3.35 stays inside the repository-owned handoff layer:

- extend `host-extension-install-package` with `installation_blueprint`;
- require `host-ui/installation-blueprint.json` in the generated file list and
  schema;
- include blueprint data in Hermes/OpenClaw host manifests generated by the
  handoff package;
- document the blueprint in Runtime Kit and host plugin/skill docs;
- prove through tests and schema drift checks that the blueprint is declarative
  only.

Do not add a new command. Do not install, sign, publish, start, bind LAN,
advertise Bonjour, mint tokens, run private adapters, write Codex user-home
plugin/skill roots, or expose anything beyond `/mobile/v1` to Kaka iPhone.

## Required User Experience

The install experience should have one normal-user path:

1. Install Kaka Mobile Bridge from the Hermes Plugin or OpenClaw Skill channel.
2. Open the host UI entry point, such as
   `Settings > Plugins > Kaka Mobile Bridge` or
   `Settings > Skills > Kaka Mobile Bridge`.
3. See a disabled-by-default bridge state after install.
4. Enable the bridge with an explicit user action.
5. Keep loopback-only as the default bind state.
6. Opt into trusted LAN and Bonjour only through visible controls.
7. Show a production QR with a short TTL and single-use pairing semantics.
8. Pair Kaka iPhone through `/mobile/v1`.
9. Run health, revoke iPhone, re-pair, update, open logs, repair port conflict,
   and uninstall from the host UI.
10. Keep provider keys, private adapter command paths, runtime store paths,
    TLS private key paths, tokens, embeddings, and raw logs off the phone.

## Artifact Boundaries

Future development must keep these surfaces separate:

- **Ordinary-user Host Extension:** the installable Hermes Plugin or OpenClaw
  Skill/sidecar package, owned and distributed by the host runtime.
- **Runtime Kit generators:** repository-owned contracts and handoff commands
  that help host teams build, validate, and release the Host Extension.
- **Codex developer automation:** optional source-only automation for host
  engineers; never the public user installer.

## Recent Non-Installation Follow-Ups

P3.25 is now implemented as the repo-owned follow-up while P3.7 waits on real
Hermes/OpenClaw package materials. It preserves completed photo-edit task
detail after reopening a store-backed bridge by storing only phone-safe result
manifest fields in runtime task metadata.

P3.25 must keep these boundaries:

- raw input/output bytes remain in `runtime_assets`;
- task metadata stores only variant ID, label, asset ID, explanation, and
  allowlisted structured recipe/status fields;
- variant `download_url` is rebuilt from `asset_id` in task detail responses
  and is not persisted;
- task lists remain summary-only;
- store-backed completed task events expose only `variant_count`, not asset IDs,
  download URLs, paths, provider endpoints, tokens, or raw payloads.

P3.26 is also implemented as a blocked-period follow-up. It reviews a local
host/runtime-owned Recall retrieval materials manifest through the existing
P3.21 readiness boundary. It can report
`accepted_for_external_retrieval_packaging_review`, but it does not fetch refs,
validate signatures, invoke providers, expose provider endpoints or keys,
include retrieval internals in Recall export, or change
`/mobile/v1/recall/search`.

## P3.19 Accepted Scope

P3.19 was accepted because it produced concrete repository-owned handoff
improvements without turning Runtime Kit or Codex automation into the ordinary
user installer:

- a stricter generated host UI acceptance file for the Kaka Mobile Bridge panel;
- an install-drill runbook that covers install, explicit enable, QR, Bonjour,
  TLS readiness, health, revoke/re-pair, update, failure recovery, logs, and
  uninstall;
- release-gate commands that point host teams to P3.6 readiness, P3.2
  conformance, P3.4 evidence manifest, and P3.18 developer automation source;
- docs that let a host engineer package Kaka without telling users to write
  adapter code, export environment variables, paste Runtime Kit commands, or
  install Codex automation.

## Anti-Goals

Do not:

- expose Hermes/OpenClaw private host APIs to Kaka iPhone;
- make `--private-adapter-command`, `HERMES_KAKA_HOST_API`, or
  `OPENCLAW_KAKA_HOST_API` the normal onboarding path;
- install Codex plugins or skills for ordinary users;
- write into `~/plugins`, `~/.codex/skills`, `~/.agents/plugins`, or marketplace
  files as part of Kaka user onboarding;
- auto-start the bridge during package install;
- bind LAN, advertise Bonjour, mint mobile tokens, create login items, or run
  private adapter commands without explicit user action;
- claim P3.7 readiness without the eight host-owned P3.6 inputs.
